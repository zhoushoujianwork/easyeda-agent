# ADR 0002: 前置设计方案书 + 分档交互模式(决策交互化,执行自动化)

## Status

Accepted. (2026-07-08)

## Decision

把「AI 先充分思考、再逐步推进」落地为两个正交机制,而**不是**「每一步都等用户确认」:

1. **S0 升级为「设计方案书(Design Proposal)」阶段** — 在放置第一个元件之前,
   产出一份覆盖**原理图 + PCB 全程**的方案书,包含所有前置架构决策
   (模块划分、电源树、页规划、叠层数、地策略、RF/天线约束、接口取向等),
   每个决策点列出**选项 + 已知坑 + 推荐方案**,一次性交用户确认。
   方案书必须产出**机器可消费的 spec**(JSON),后续阶段(S2 分区、P4 禁布区、
   P8 叠层/地策略)从 spec 读取执行,使「先想清楚」成为可执行约束而非装饰文档。

2. **交互分三档**,写入 skill,由用户意图/场景选档:

   | 模式 | 确认点 | 场景 |
   |---|---|---|
   | 全自动(auto) | 0(仅破坏性操作确认) | 回归基准、CI、ClawFlow operator、录制脚本 |
   | 里程碑确认(milestone,真实用户默认) | ① S0 方案书 ② 原理图完成转 PCB 前 ③ 发板/交付前 | 正常客户设计 |
   | 逐步确认(step) | 每个 S/P 阶段 | 教学、演示、用户显式要求 |

3. **坑分两类处置**,判据:*用户的回答会改变做法的才问,不会改变的不问。*
   - **Guardrail(永不问用户)**:有唯一正确答案的坑 — save 纪律、mutation 后
     `doc reload`、layout-lint/DRC 硬门、PLANE 后禁异网 via、pour-rebuild 时机等。
     继续以硬门禁形式内置在流程里。
   - **Decision(进方案书)**:真实权衡 — 2 层 vs 4 层成本、单/双取向 USB
     (A6+B6/A7+B7 tie)、单 GND PLANE vs 分区 pour 桥地、天线 keepout 范围、
     选型成本档位等。以「选项+坑+推荐」形式呈现,由用户拍板。

## Context

原有 `design-flow.md` 已是分阶段+硬门禁流水线(S0–S6 + P0–P10),原则 7 也已
支持 opt-in 的逐步确认,所以「AI 闷头干」的问题只对了一半。真正的短板是:

- **S0 预分析太浅且只看原理图。** 叠层、地策略、天线 keepout、USB 取向等 PCB
  架构决策要走到 P4/P8 才被想到,而实测最贵的返工恰恰来自决策后置
  (天线禁布区后置逼 BLINK 重绕;地策略选错要重铺)。PCB 层面的架构决策必须在
  画第一笔原理图之前定下来。
- **坑全是内部 guardrail,用户看不到选择。** 官方板对标(N8R8/实战派)沉淀的
  判据(地策略判据、双取向 USB)本质是权衡而非规则,不同产品定位选法不同,
  但现在被写死成默认值,用户没有参与决策的入口。

同时,「每步都停下来让用户确认」若做成**默认**行为,会摧毁项目最值钱的两个能力:

- **固定测试用例失效**:CLAUDE.md 首要准则要求 `esp32MiniRequire.md` 端到端
  无人跑通作为回归基准;每步阻塞则回归跑不动。
- **无人值守场景失效**:ClawFlow operator、录制/演示脚本都要求无阻塞执行。
- **确认疲劳**:17 个阶段中绝大多数只有一个正确答案,问了不产生信息,只产生
  噪音;几轮后用户盲点确认,交互形同虚设。

因此核心裁决是:**决策交互化、执行自动化** — 把用户的参与集中在少量高价值的
前置决策点上,执行段保持机械门禁 + 全自动。

## Consequences

- **主要是 Skill 层改动**(符合「Skill 优先」准则):
  - `references/design-flow.md`:S0 重写为方案书阶段;新增三档交互模式说明;
    原则 7 归并进 step 档。
  - 新增 `references/design-decisions.md`(决策点清单):从官方板对标判据、
    USB 取向、地策略、叠层选择等 memory/conventions 中沉淀,每条含
    选项、坑、推荐、判据来源。
  - 方案书 spec 的 schema 约定(模块/页规划/叠层/地策略/禁布区),
    与既有 `sch autolayout --spec` 思路对齐,后续阶段引用而非重新发明。
- **底层(Go CLI)改动最小**:可能仅需 spec schema 校验或 `easyeda plan`
  类辅助命令;不新增交互能力到 daemon——交互由 agent(skill 编排)承担。
- **回归基准不变**:esp32MiniRequire 用例以全自动档照跑;新增一次人工
  里程碑档走查验证交互体验。
- **录制/演示模式不受影响**:仍是数据门+可视化产物门,叠加在任一交互档上。
- 反面约束:方案书**不得**退化为写完即弃的长文——凡是方案书里定的决策,
  执行阶段必须从 spec 读取;执行与方案不一致视为 bug。
